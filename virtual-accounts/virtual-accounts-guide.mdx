Virtual Accounts (VAs) are dedicated fiat bank accounts assigned to each customer for receiving payments. Funds deposited to a VA are automatically linked to that customer's deposits in the Coinut system.

## What is a Virtual Account?

A Virtual Account provides a unique set of banking details for each customer:

- Customer sends fiat to their dedicated VA
- Coinut detects the incoming transfer
- Funds are automatically associated with the correct customer
- No manual reconciliation needed

## Supported Currencies

There's a single `bicSwift` field on the detail response for every currency — it carries the BSB number for AUD and the SWIFT/BIC code for everything else. There's no separate `bsb` or `swift` field, and no `sortCode` field.

| Currency | Account Format | Key Fields | Geographic Focus |
| --- | --- | --- | --- |
| AUD | BSB \+ Account Number | `payId`, `bicSwift` (BSB), `accountNumber` | Australia |
| EUR | IBAN | `iban`, `bicSwift` (SWIFT/BIC) | Europe |
| GBP | IBAN | `iban`, `bicSwift` (SWIFT/BIC) | United Kingdom |
| SGD / USD | IBAN | `iban`, `bicSwift` (SWIFT/BIC) | Enabled in the current create schema; confirm corridor-specific receiving details with Coinut before shipping UX |

## VA Lifecycle

```text
PENDING → VERIFIED (after compliance review)
```

A VA starts in `PENDING` status after creation. Once compliance review completes, it transitions to `VERIFIED`. Deposits can typically be received while `PENDING`, but may be held until verification completes.

## Creating VAs

Call [`POST /virtual-account/create`](/api-reference/virtual-account/create-a-fiat-virtual-account) <Icon icon="link" size={14} />. See the API Reference for the full request schema.

```json
{
  "customerId": "cust_abc123",
  "currency": "AUD",
  "whitelist": {
    "bsbNumber": "062000",
    "accountNumber": "12345678",
    "accountName": "Alice Smith"
  }
}
```

<Note>
  [`POST /virtual-account/create`](/api-reference/virtual-account/create-a-fiat-virtual-account) <Icon icon="link" size={14} /> itself only returns the new VA's `id`. Immediately call [`GET /virtual-account/detail`](/api-reference/virtual-account/get-fiat-va-detail) <Icon icon="link" size={14} /> to get the full banking details — see the API Reference for the response schema.

## AUD Virtual Accounts & PayID

PayID is an email-address-style identifier for instant bank transfers in Australia. It replaces the need to share BSB and account numbers.

- **Format**: `john.doe@example.com`
- **Speed**: Near-instant (via NPP/Osko)
- **Availability**: 24/7, including weekends

### Display Format

```text
PayID: john.doe@example.com
BSB: 062000
Account: 12345678
```

## EUR/GBP Virtual Accounts

For European and UK customers, VAs use IBAN format:

### EUR (SEPA)

```text
IBAN: DE89370400440532013000
SWIFT/BIC: COINUTDEFF
```

- Supports SEPA transfers
- Settlement typically 1 business day
- Available for EU/EEA customers

### GBP (Faster Payments / BACS)

```text
IBAN: GB29NWBK60161331926819
SWIFT/BIC: COINUTGB2L
```

<Note>
  There's no separate `sortCode` field — only `iban` and `bicSwift` are returned for GBP accounts.
</Note>

- Supports Faster Payments (near-instant, up to GBP 1M)
- Supports BACS (3 business days)
- Settlement typically 1 business day

## Whitelist Management (AUD Only)

AUD virtual accounts support a whitelist that restricts which bank accounts can send funds. This adds an extra layer of compliance control.

| Endpoint | Method | Description |
| --- | --- | --- |
| [`/virtual-account/create-aud-whitelist`](/api-reference/virtual-account/create-a-whitelist-entry-for-an-aud-va) <Icon icon="link" size={14} /> | `POST` | Create a new whitelist for an AUD VA |
| [`/virtual-account/update-aud-whitelist`](/api-reference/virtual-account/update-a-whitelist-entry-for-an-aud-va) <Icon icon="link" size={14} /> | `POST` | Update an existing whitelist |
| [`/virtual-account/get-aud-whitelist`](/api-reference/virtual-account/get-whitelist-entries-for-an-aud-va) <Icon icon="link" size={14} /> | `GET` | Retrieve the current whitelist |

### Whitelist Format

Creating and updating a whitelist entry take slightly different `sourceAccount` shapes:

```json Create
{
  "id": "va_123",
  "sourceAccount": {
    "bsbNumber": "062000",
    "accountNumber": "12345678",
    "accountName": "Alice Smith"
  }
}
```

```json Update
{
  "id": "va_123",
  "sourceAccount": {
    "id": 1,
    "bsbNumber": "062000",
    "accountNumber": "12345678",
    "accountName": "Alice Smith",
    "accountStatus": "active"
  }
}
```

<Warning>
  The virtual account identifier field is `id`, not `vaId` — submitting `vaId` will fail validation since `id` is required. On create, only `bsbNumber` and `accountNumber` are required (`accountName` is optional). On **update**, `bsbNumber`, `accountNumber`, `accountName`, **and `accountStatus` are all required** — omitting `accountStatus` will fail validation.
</Warning>

When a whitelist is active, deposits from non-whitelisted accounts may be rejected or held for manual review.

<Note>
  [`POST /virtual-account/create-aud-whitelist`](/api-reference/virtual-account/create-a-whitelist-entry-for-an-aud-va) <Icon icon="link" size={14} /> and [`.../update-aud-whitelist`](/api-reference/virtual-account/update-a-whitelist-entry-for-an-aud-va) <Icon icon="link" size={14} /> return their `data` as a **single** whitelist entry object — only [`GET /virtual-account/get-aud-whitelist`](/api-reference/virtual-account/get-whitelist-entries-for-an-aud-va) <Icon icon="link" size={14} /> returns an **array** of entries. `sourceAccount.id` is optional on update, but omitting it sends `0` as the entry identifier, so always include it once you have it.
</Note>

## Best Practices

- **One VA per customer per currency**. Don't create multiple VAs for the same customer\+currency pair.
- **Store VA details** in your database alongside the customer record for quick lookups.
- **Display banking details clearly** with copy-to-clipboard buttons in your UI.
- **Generate QR codes** for PayID to simplify mobile banking.
- **Monitor `DEPOSIT_RECEIVED`** webhook for incoming funds — don't rely on polling alone.
- **Handle `ACTION_REQUIRED` deposits** — manual follow-up may be needed for some incoming transfers.

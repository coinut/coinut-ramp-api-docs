This page walks through the full onramp journey end to end, at a glance â€” from creating a customer to confirming on-chain settlement. Each step links to the page with full details, request/response fields, and edge cases. Offramp follows the mirror sequence â€” see [Offramp Integration](offramp-integration).

<Steps>
  <Step title="Create Customer">
    Call [`POST /customer/create`](api/customer-create) with name, email, `type="DEPOSITOR"`, and `isIndividual=true`.

    ```json
    {
      "name": "John Doe",
      "email": "john@example.com",
      "type": "DEPOSITOR",
      "isIndividual": true,
      "externalId": "user_12345"
    }
    ```

    See [Customer Lifecycle](customer-lifecycle) for the full field reference.
  </Step>
  <Step title="Wait for KYC Approval">
    Listen for the `CUSTOMER_APPROVED` webhook event. You can also poll [`GET /customer/detail`](api/customer-detail) to check status.

    Customer status transitions: `PENDING` â†’ `CONFIRMED` | `REJECTED` (a customer can also end up `DELETED`). See [Customer Lifecycle](customer-lifecycle).
  </Step>
  <Step title="Add Crypto Address">
    Call [`POST /crypto-address/add`](api/crypto-address-add) with `customerId`, `currency` (`USDT` or `USDC`), `address`, `label`, and optionally `network` (`ETH`, `TRX`, or `MATIC`).

    Store the returned `id` â€” you'll need it for the autoramp flow. See [Crypto Addresses](crypto-addresses).
  </Step>
  <Step title="Create Onramp Flow">
    Call [`POST /autoramp-flow/onramp`](api/autoramp-onramp) with the fiat source and crypto destination:

    ```json
    {
      "from": {
        "customerId": "cust_abc123",
        "currency": "AUD"
      },
      "to": "crypto_address_id_from_step_3"
    }
    ```

    This creates a virtual account automatically and links its deposits to automatic crypto conversion. The response only contains the new flow's `id` â€” see [Autoramp Flows](autoramp-flows).
  </Step>
  <Step title="Display Virtual Account to Customer">
    The flow response doesn't include the virtual account's banking details directly â€” fetch [`GET /autoramp-flow/detail`](api/autoramp-detail) and read them from `fromVirtualAccount`. There's a single `bicSwift` field for every currency â€” it carries the BSB number for AUD and the SWIFT/BIC code for everything else. There's no separate "sort code" field.

    - **AUD**: Display `payId`, `bicSwift` (BSB), and `accountNumber`
    - **EUR / GBP / SGD / USD**: Display `iban` and `bicSwift` (SWIFT/BIC)

    See [Virtual Accounts](virtual-accounts-guide) and [QR Code Payments](qr-code-payments) for generating a scannable PayID QR.
  </Step>
  <Step title="Monitor Deposits">
    Listen for `DEPOSIT_RECEIVED` (funds detected) and `DEPOSIT_APPROVED` (funds cleared), or poll [`GET /deposit/list`](api/deposit-list) with the customer ID. See [Deposits](deposits).
  </Step>
  <Step title="Track Trade">
    After deposit approval, listen for `TRADE_CREATED` (trade initiated) and `TRADE_SETTLED` (trade complete, crypto sent), or poll [`GET /trade/list`](api/trade-list). See [Trades](trades).
  </Step>
  <Step title="Confirm Settlement">
    Once the crypto has been broadcast, the `TRADE_SETTLED` payload includes a nested `withdrawal.txId` â€” use that to verify the transaction on-chain:

    - ERC-20: `https://etherscan.io/tx/{withdrawal.txId}`
    - TRC-20: `https://tronscan.org/#/transaction/{withdrawal.txId}`

    `withdrawal` may be absent on the first delivery of `TRADE_SETTLED` if the broadcast hasn't completed yet â€” treat it as optional. There is no top-level `txHash` field. See [Trades](trades).
  </Step>
</Steps>

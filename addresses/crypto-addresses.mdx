---
title: "Crypto Addresses"
description: "Registering customer crypto addresses and understanding deposit addresses."
---

Two distinct address types cover the crypto side of a flow: **crypto addresses** (a customer's own wallet — the exit point for onramp) and **deposit addresses** (a Coinut-issued receiving address — the entry point for offramp). They are separate resources with separate lifecycles.

## Crypto Addresses (Onramp Exit)

A crypto address is the customer's own wallet — where converted crypto is sent after an onramp deposit settles.

Call [`POST /crypto-address/add`](/api-reference/crypto-address-add) <Icon icon="link" size={14} />:

```json
{
  "customerId": "e2f8c1d4-3b7a-4c19-9f6e-1a2b3c4d5e6f",
  "currency": "USDT",
  "address": "0xA1b2C3d4E5f6789012345678901234567890aBcD",
  "label": "Main Wallet",
  "network": "ETH"
}
```

| Field | Required | Description |
| --- | --- | --- |
| `customerId` | Yes | Must belong to a `CONFIRMED` customer |
| `currency` | Yes | `USDT` or `USDC` |
| `address` | Yes | Destination wallet address |
| `label` | Yes | Display label |
| `network` | No | `ETH` (ERC-20), `TRX` (TRC-20), or `MATIC` (Polygon) |

Crypto addresses have **no approval step** — there is no `status` field on this resource, unlike deposit addresses below. The address is usable immediately after creation.

See [`GET /crypto-address/list`](/api-reference/crypto-address-list) <Icon icon="link" size={14} /> and [`GET /crypto-address/detail`](/api-reference/crypto-address-detail) <Icon icon="link" size={14} /> to look up existing addresses.

## Deposit Addresses (Offramp Entry)

A deposit address is a Coinut-generated receiving address — where a customer sends crypto to trigger an offramp. It's created automatically when you call [`POST /autoramp-flow/offramp`](/api-reference/autoramp-offramp) <Icon icon="link" size={14} /> (see [Autoramp Flows](/transactions/autoramp-flows)), or directly via [`POST /deposit-address/create`](/api-reference/deposit-address-create) <Icon icon="link" size={14} />.

Unlike crypto addresses, deposit addresses go through an **approval step** and carry a `status`:

| Status | Meaning |
| --- | --- |
| `PENDING` | Just created — `address` is **not yet populated** |
| `VERIFIED` | Approved — `address` is populated and safe to display |
| `REJECTED` | Rejected — do not display, contact support |
| `DELETED` | Removed |

<Warning>
  A newly created deposit address starts `PENDING` with `address: null`. It is **not** available synchronously — poll [`GET /deposit-address/detail`](/api-reference/deposit-address-detail) <Icon icon="link" size={14} /> or listen for the `DEPOSIT_ADDRESS_APPROVED` webhook, and only show the address to the customer once `status` leaves `PENDING` and `address` is populated.
</Warning>

Supported networks: `ETH` and `TRX` today. `MATIC` is accepted by the schema but not yet supported end-to-end — don't offer it to customers.

## API Playground vs. Your Integration

<Warning>
  The interactive **API Playground** on this site (the "Try it" panel on each endpoint page) behaves differently from calling the API from your own service — don't treat a successful Playground call as proof that your own signing implementation is correct.
</Warning>

The Playground is useful for a first check that your **API key and secret are valid** and for exploring request/response shapes, but the signing step is not representative of a real integration:

| | API Playground | Your Service (Sandbox / Production) |
| --- | --- | --- |
| Who computes the HMAC-SHA256 signature | The Playground's own backend, using the API key/secret you paste into the docs UI | Your service, using the canonical-string algorithm described in the Authentication page |
| What it proves | Your API key/secret pair is valid and active | Your actual signing implementation is correct end-to-end |
| Where credentials are entered | Directly into the browser-based Playground form | Held server-side in your own environment |

Once you've confirmed your API key and secret work in the Playground, move on to implementing the signature yourself — see the [Authentication](/getting-started/authentication) page for the full canonical string spec and code examples — and test that implementation directly against the sandbox base URL (`https://ramp-sandbox.hexarails.ai`).

## Authentication

Every request must be authenticated using **HMAC-SHA256** request signing. You receive an API key and secret during onboarding. See the [Authentication](/getting-started/authentication) page for the full canonical string format, language-specific code examples, and best practices.

In short: include `X-API-Key`, `X-Timestamp`, `X-Nonce`, and `X-Signature` headers on every request. The signature is built from a newline-delimited canonical string (`METHOD\nHOST\nPATH\nQUERY\nBODY_SHA256_HEX\nTIMESTAMP\nNONCE`) signed with your API secret.

<Note>
  `X-Timestamp` is included in the signed canonical string but is **not currently checked against server time** — there is no replay-window enforcement today. Keep your server clock accurate with NTP anyway: it's required for audit trails, and a staleness check may be added in the future without notice.
</Note>


## Request & Response Format

- All requests and responses use **JSON** with **UTF-8** encoding.
- Set `Content-Type: application/json` on every request.
- Use **POST** for mutations and **GET** for reads unless specified otherwise.

### Success Response

All responses use this envelope:

```json
{
  "status": "SUCCESS",
  "data": { }
}
```

<Note>
  A small set of older endpoints return their data object directly without the `{status, data}` wrapper. All current endpoints follow the envelope above — see the API Reference for per-endpoint response schemas.
</Note>

### Error Response

Errors are returned in a consistent envelope. The `status` field indicates the error category, and `message` provides a human-readable description.

```json
{
  "status": "INVALID_PARAMETERS | UNAUTHORISED | FAILED",
  "message": "Human-readable description of the error"
}
```

| `status` | Meaning |
| --- | --- |
| `INVALID_PARAMETERS` | Missing or invalid request parameters (validation failed) |
| `UNAUTHORISED` | Authentication failed — bad signature, API key, timestamp, or nonce |
| `FAILED` | Internal or downstream processing error |

## HTTP Status Codes

The API returns three HTTP status codes in normal operation. There is no `429` or `500` — every application-level failure is reported as `422` with a `status`/`message` body.

| Code | Meaning |
| --- | --- |
| `200` | Request succeeded On a small set of older endpoints, check the JSON `status` field rather than relying on the HTTP code alone |
| `401` | Authentication failed — see [Authentication](/getting-started/authentication) |
| `422` | Request was authenticated but invalid, or a downstream operation failed |

A `403` is never returned by the API itself — if you see one, it was blocked by infrastructure (WAF/gateway/IP block) before reaching the application.

## Rate Limiting

<Note>
  The Ramp application itself does not implement per-key rate limiting, a `429` response, or a `Retry-After` header today — any throttling in front of the API would be enforced at the infrastructure/gateway level, outside this service. Don't build client logic that depends on a specific limit or on receiving `429`; contact support if you're planning a high-volume integration so current infrastructure limits can be confirmed for your account.
</Note>

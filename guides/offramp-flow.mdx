## Use Case

Enable customers to offramp **cryptocurrency into fiat** (`USDT`, `USDC`). This works by setting up an autoramp rule that automatically converts crypto received at a deposit address and pays it out to a bank account.

Examples:

- Set up a **deposit address** which converts USDT on TRX into EUR and pays out to a Deutsche Bank account
- Set up a **deposit address** which converts USDC on Ethereum into AUD and pays out to a personal bank account

## Example Flow

<Steps>
  <Step title="Set up the flow">
    Create an offramp autoramp flow for a customer, linking a new USDT/TRX deposit address to the customer's EUR bank account.
  </Step>
  <Step title="Customer sends crypto">
    Customer sends 1,000 USDT to the deposit address on the specified network.
  </Step>
  <Step title="Coinut converts">
    Coinut detects the deposit, confirms it, and executes the trade automatically.
  </Step>
  <Step title="Fiat is queued">
    ~998 EUR is queued for payout to the customer's bank account.
  </Step>
  <Step title="Repeatable">
    The customer can send again at any time â€” the deposit address and flow are persistent. Non-supported assets sent to the address are returned to sender.
  </Step>
</Steps>

## Prerequisites

Every step must complete before moving to the next.

<Steps>
  <Step title="Customer is CONFIRMED">
    The customer must have completed KYC. See [Customer Lifecycle](/customers/customer-lifecycle).
  </Step>
  <Step title="Register the bank account">
    Register the destination bank account via [Bank Accounts](/destinations/bank-accounts). You need its `id` before creating the flow.
  </Step>
  <Step title="(Optional) Preview the rate">
    Call [`GET /autoramp-flow/trade-preview`](/api-reference/autoramp-flow/trade-preview) <Icon icon="link" size={14} /> with `fromCurrency`, `toCurrency`, and `amount` for an indicative rate. This is a live estimate, not a locked quote â€” the flow always executes at the rate current when the deposit clears.
  </Step>
</Steps>

## Implementation

Create the offramp flow with [`POST /autoramp-flow/offramp`](/api-reference/autoramp-flow/create-an-offramp-flow-with-deposit-address) <Icon icon="link" size={14} />. The deposit address is created (or reused, for an existing customer/currency/network combination) automatically from `from`.

### Response

```json
{
  "status": "SUCCESS",
  "data": {
    "id": "uuid-12345"
  }
}
```

The response only contains the new flow's `id` â€” not the deposit address. Fetch [`GET /autoramp-flow/detail`](/api-reference/autoramp-flow/get-autoramp-flow-detail) <Icon icon="link" size={14} /> to retrieve the created deposit address under `fromDepositAddress`.

<Warning>
  The deposit address starts `PENDING` with `address: null` â€” it is **not** available synchronously. Poll [`GET /deposit-address/detail`](/api-reference/deposit-address/get-deposit-address-detail) <Icon icon="link" size={14} /> or wait for the `DEPOSIT_ADDRESS_APPROVED` webhook before displaying it to the customer. See [Deposit Addresses](/deposit-rails/deposit-addresses).
</Warning>

### Error response

```json
{
  "status": "INVALID_PARAMETERS",
  "message": "invalid parameters"
}
```

## Monitor via Webhooks

Subscribe to `DEPOSIT_ADDRESS_APPROVED`, `DEPOSIT_RECEIVED`, `DEPOSIT_APPROVED`, `TRADE_CREATED`, and `TRADE_SETTLED` rather than polling. See [Webhooks](/webhooks/overview).

<Note>
  `TRADE_SETTLED` confirms the crypto-to-fiat trade executed â€” it does **not** confirm the bank transfer landed. There is no separate webhook for offramp bank-transfer completion; reconcile actual bank arrival through your own banking feed. See [Trades](/transactions/trades).
</Note>

## Overview

Coinut sends webhook events to your registered callback URL when important events occur.

All webhooks:

- Use the `POST` method
- Send a JSON body using the `{event, payload}` envelope
- Are only sent for event types you've subscribed to, and only once you have a callback URL configured (see [Setting Up Webhooks](#setting-up-webhooks))
- Are signed with the same HMAC-SHA256 scheme used for incoming requests (see [Webhook Signature Verification](/webhooks/signature-verification))
- Accept any `2xx` response as a successful delivery — respond as soon as you've durably queued the event

## Setting Up Webhooks

<Steps>
  <Step title="Register callback URL">
    Send [`POST /callback-url`](/api-reference/common/update-callback-url) <Icon icon="link" size={14} /> with your HTTPS endpoint.

    ```json
    {
      "callbackUrl": "https://your-domain.com/webhooks/coinut"
    }
    ```
  </Step>
  <Step title="Subscribe to events">
    Send [`POST /subscribed-events`](/api-reference/common/update-subscribed-events) <Icon icon="link" size={14} /> with the event types you want to receive.

    ```json
    {
      "events": [
        "CUSTOMER_APPROVED",
        "DEPOSIT_RECEIVED",
        "TRADE_SETTLED"
      ]
    }
    ```

    <Note>
      Both steps are required — an event is only queued for delivery if you have a callback URL **and** that event type appears in your subscribed events. Missing either one means the event is silently skipped.
    </Note>
  </Step>
  <Step title="Verify signature">
    Every webhook is signed exactly like an incoming API request (see [Authentication](/getting-started/authentication)): the same four headers (`X-API-Key`, `X-Timestamp`, `X-Nonce`, `X-Signature`) are sent with the callback, built from the same HMAC-SHA256 canonical string. Verify it before trusting the payload — see [Webhook Signature Verification](/webhooks/signature-verification) for the exact construction.
  </Step>
  <Step title="Acknowledge receipt">
    Return any `2xx` status as soon as you've durably stored or queued the event. Process asynchronously — failed deliveries are retried automatically (see [Retries](/webhooks/retries)), so your handler must be idempotent.
  </Step>
</Steps>

## Event Types

Available event types are fully documented in the `WebhookEvent` schema in the [API Reference](/api-reference) — including every valid `event` enum value and the payload schema for each type. The following domains are covered:

| Domain | Events |
| --- | --- |
| Customer | `CUSTOMER_APPROVED`, `CUSTOMER_REJECTED`, `CUSTOMER_DELETED` |
| Virtual Account | `VIRTUAL_ACCOUNT_APPROVED`, `VIRTUAL_ACCOUNT_REJECTED` |
| Deposit Address | `DEPOSIT_ADDRESS_APPROVED`, `DEPOSIT_ADDRESS_REJECTED` |
| Deposit | `DEPOSIT_RECEIVED`, `DEPOSIT_APPROVED`, `DEPOSIT_REJECTED` |
| Trade | `TRADE_CREATED`, `TRADE_SETTLED` |
| Payment | `PAYMENT_SETTLED` |

## Best Practices

- **Always verify signatures.** Validate every delivery against the [Webhook Signature Verification](/webhooks/signature-verification) scheme before processing the payload.
- **Handle idempotently.** See [Retries](/webhooks/retries) — your handler must tolerate duplicate deliveries safely.
- **Respond quickly, process asynchronously.** Acknowledge with a `2xx` as soon as the event is durably queued.
- **Log everything.** Store raw payloads and headers for debugging.
- **Expect retries.** The same event occurrence can be redelivered up to 5 times — see [Retries](/webhooks/retries).

## Testing Webhooks Locally

<Tip>
  Test webhooks locally with [ngrok](https://ngrok.com). Run `ngrok http 3000`, then register the HTTPS tunnel URL as your callback endpoint. Coinut will deliver events to your local machine.
</Tip>
